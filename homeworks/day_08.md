# Токены, стоимость и переполнение контекста (CLI-демо)

🔥 День 8. Подсчет токенов в диалогах

Цель: показать, что агент умеет считать токены:
👉 для текущего запроса  
👉 для всей истории диалога  
👉 для ответа модели  

И как меняются токены/стоимость и поведение агента при росте диалога и при переполнении лимита контекста модели.

---
## Что реализовано

- В CLI появился флаг `--tokens`, который печатает breakdown токенов после каждого запроса.
- В `--chat` режиме можно использовать команду `@tokens`.
- `@tokens` / `@tokens on` включает печать токенов.
- `@tokens off` выключает печать токенов (чтобы не раздувать стоимость/запросы).
- breakdown печатается в формате `current_request_tokens`, `dialog_history_tokens`, `response_model_tokens`.
- `current_request_tokens` — токены текущего user-сообщения, без system.
- `dialog_history_tokens` — токены всех предыдущих сообщений диалога, без system.
- `response_model_tokens` — completion_tokens, сколько модель сгенерировала.
- Дополнительно печатаются `prompt_tokens_total`, `completion_tokens` и `total_tokens` (как вернул провайдер).

Технически: токены `prompt_tokens` и `completion_tokens` берутся из `usage` ответа провайдера, а разделение “история vs текущий запрос” делается через сравнительную оценку:
агент делает дополнительный dry-run-запрос, чтобы получить токены для `system + история` и вычесть вклад system-only.

---
## Как запустить демо

Из корня проекта `hw01/`:

1) Убедиться, что `.env` содержит ключ провайдера (пример — OpenRouter):
```bash
cat > .env <<'EOF'
OPENROUTER_API_KEY=ваш_api_ключ
EOF
```

2) Включить и токены, и стоимость/latency (стоимость печатается из `usage.cost` в режиме `-v`):
```bash
python3 llm_cli.py --chat --tokens -v
```

Подсказка по логам:
- ответ модели печатается в `stdout` (после `agent>`)
- `[stats] ...` и `[tokens] ...` печатаются в `stderr` (удобно объединить в один лог: `2>&1`)

Для удобства логирования используйте:
```bash
python3 llm_cli.py --chat --tokens -v 2>&1 | tee tokens_demo.log
```

## Эксперимент 1: короткий диалог

Цель: увидеть, что `dialog_history_tokens` почти не растут, а `current_request_tokens` примерно соответствует размеру текущего сообщения.

Пошагово:

1) Запуск:
```bash
python3 llm_cli.py --chat --tokens -v
```
2) Диалог (пример):
```text
you> Суммаризируй текст: "A" — это буква, "B" — это звук. Напиши 2 предложения.
agent> ...
you> Теперь объясни разницу между буквой и звуком на 1 примере.
agent> ...
you> exit
Bye!
```

Что должно быть видно в выводе:
- `prompt_tokens` / `total_tokens` (в `[stats]`) после 2-го сообщения увеличатся
- `dialog_history_tokens` во 2-м сообщении будет больше, чем в 1-м (потому что в запрос уже попадет одно предыдущее user/assistant сообщение)
- `response_model_tokens` будет зависеть от того, насколько длинный ответ вы запросили

## Эксперимент 2: длинный диалог

Цель: увидеть рост “история диалога -> prompt_tokens_total -> стоимость”.

Пошагово:

1) Запуск (оставляем те же флаги и параметры модели):
```bash
python3 llm_cli.py --chat --tokens -v
```
2) Диалог (пример “длинного” цикла, 8-12 сообщений):
```text
you> Напиши короткий план действий для задачи: "Сделать скрипт логирования токенов". 5 пунктов.
agent> ...
you> Теперь уточни каждый пункт: по 2 предложения на пункт.
agent> ...
you> Дай примеры входов/выходов и ограничения по формату.
agent> ...
you> Повтори это в другом стиле (но не меняй смысл).
agent> ...
you> Добавь 3 возможных ошибки пользователя и как их обработать.
agent> ...
you> exit
Bye!
```

Ожидаемое наблюдение:
- `dialog_history_tokens` будет расти от сообщения к сообщению, т.к. в payload уходит вся история из SQLite
- `prompt_tokens_total` и `total_tokens` будут расти в основном из-за history
- `response_model_tokens` может расти медленнее (или оставаться примерно одинаковым), если вы держите `--max-output-tokens` / требования к длине ответа стабильно

Про стоимость:
- если провайдер возвращает `usage.cost`, это видно в `[stats] ... cost=...`
- в режиме `--tokens` возможны дополнительные dry-run запросы для оценки breakdown (`history vs current`), поэтому стоимость может быть чуть выше, чем “без подсчета”

## Эксперимент 3: переполнение лимита контекста

Цель: показать, что происходит, когда `system + вся история + текущий запрос` становится больше контекста модели.

Как воспроизвести:

Вариант A (быстро, через очень длинный ввод):
```text
you> Отвечай, как обычно, но перед ответом ОБЯЗАТЕЛЬНО обработай мой текст ниже.
Сгенерируй в этом сообщении большой массив повторяющегося текста (например, 20-50 тыс. символов).
...
```

Вариант B (через длинную историю):
1) держать чат открытым и каждый раз добавлять длинные сообщения (или несколько абзацев)
2) дождаться момента, когда провайдер вернет ошибку переполнения контекста

Ожидаемое “что ломается”:
- запрос может завершиться ошибкой провайдера (часто это HTTP 400/413 с текстом вроде “context length exceeded” / “maximum context length”)
- в нашем CLI это проявится как: приложение напечатает `HTTP error ...` или сообщение провайдера; интерактивный процесс завершится, потому что ошибка не перехватывается внутри цикла
- дополнительно: если вы “перетягиваете” ограничения output tokens, иногда провайдер может вернуть пустой `message.content`, и тогда агент упадет с `Received empty assistant content ...`

Что показать в отчете:
- строку/фрагмент сообщения об ошибке из лога
- как в предыдущих шагах рос `prompt_tokens_total` до отказа

## Где в коде видно выполнение требования

- Печать breakdown токенов: `app/cli_utils.py` (`--tokens`, `print_token_stats(...)`).
- Печать breakdown токенов: `app/cli.py` (обработка `@tokens` и вызов `print_token_stats(...)`).
- Счёт токенов в агенте: `app/agent.py` (флаг `options.count_tokens`).
- Счёт токенов в агенте: `app/agent.py` (чтение `prompt_tokens_total`, `completion_tokens`, `total_tokens` из `usage`).
- Разделение “текущий запрос vs история”: `app/agent.py` (оценка `current_request_tokens` и `dialog_history_tokens` через dry-run токенов).
