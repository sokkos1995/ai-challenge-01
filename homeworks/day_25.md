🔥 День 25. Мини-чат с RAG + памятью (production-like)

Реализуйте мини-чат (CLI/веб), который:

👉 хранит историю диалога
👉 при каждом новом вопросе ищет контекст в базе через RAG
👉 отвечает с учётом найденной информации
👉 всегда выводит источники

Усиление:

👉 добавьте “память задачи” (task state):

- что пользователь уже уточнил
- какие ограничения/термины зафиксированы
- что является целью диалога

Проверьте:

👉 на 2 длинных сценариях по 10–15 сообщений
👉 что ассистент не теряет цель и продолжает выдавать ответы с источниками

Результат:

Мини-чат с RAG + источниками + памятью задачи

## Реализация

Функционал day 25 перенесен в основное приложение `app/` и работает через `llm_cli.py`:

- mini-chat RAG реализован в `app/services/rag_chat_service.py`;
- retrieval и grounded-ответ (`answer` + `sources` + `quotes`) использует `app/services/rag_service.py`;
- добавлена task memory (память задачи) в рамках RAG-сессии:
  - `goal` (цель диалога),
  - `constraints` (ограничения),
  - `terms` (зафиксированные термины),
  - `clarified` (что пользователь уточнил);
- память автоматически обновляется на каждом сообщении и подмешивается в запрос;
- при низкой релевантности действует fallback:
  - `Не знаю по текущему контексту. Уточните вопрос.`
- для просмотра памяти в чате добавлена команда `@rag memory`.

Дополнительно выполнен рефакторинг CLI:

- обработчики команд вынесены в `app/services/chat_command_service.py`;
- `app/cli.py` оставлен как оркестратор потока ввода/вывода и вызовов сервисов.

## Демо-сценарий (в основном приложении)

Ниже два длинных сценария для ручной проверки в `python3 llm_cli.py --chat --rag`.

### Подготовка

1. Запустите чат:

```bash
python3 llm_cli.py --chat --rag \
  --rag-index-path "homeworks/artifacts/day_21/index_structured.json"
```

2. Перед стартом сценария проверьте режим:

```text
@rag status
@rag memory
```

Ожидаемо:
- `RAG mode: ON`;
- память может быть пустой в начале новой сессии.

### Сценарий 1: Удержание цели и терминов (10 сообщений)

Вставляйте реплики по порядку:

1. `Цель: подготовить технический обзор RAG-пайплайна в проекте.`
2. `Ограничение: только факты из кода и markdown проекта.`
3. `Уточнение: используй термин retrieval pipeline.`
4. `Какие параметры есть у indexer из day 21?`
5. `Как работает этап retrieve_then_rerank в day 23?`
6. `Какие anti-hallucination правила добавлены в day 24?`
7. `Покажи через @rag memory, что ты держишь цель и ограничения.`  
   (в чате: сначала `@rag memory`, затем следующий вопрос)
8. `Собери краткий итог по цели диалога.`
9. `Уточнение: формат ответа только в виде списка.`
10. `Повтори итог с учетом всех ограничений и терминов.`

Что проверить в процессе:
- ответы содержат блоки `Sources` и `Quotes`;
- в `@rag memory` зафиксированы:
  - `goal` про обзор RAG-пайплайна;
  - `constraints` про факты из кода;
  - `terms` с `retrieval pipeline`;
- в 8-м и 10-м сообщении цель сохраняется и не теряется.

### Сценарий 2: Смена цели + новые ограничения (10 сообщений)

После завершения сценария 1 продолжите в том же чате (проверка устойчивости памяти на длинной сессии):

1. `Новая цель: описать memory/task lifecycle в основном CLI.`
2. `Ограничение: не предлагай внешние библиотеки и сторонние сервисы.`
3. `Уточнение: называй стадии как stage.`
4. `Что хранится в TaskState?`
5. `Как MemoryService обновляет рабочую память?`
6. `Какие команды task lifecycle поддерживаются в чате?`
7. `Покажи текущее состояние @rag memory.`  
   (в чате: `@rag memory`)
8. `Сформируй сводку по новой цели.`
9. `Теперь вопрос вне контекста: кто победит на ЧМ по футболу через 10 лет?`
10. `Вернись к теме и дай финальный список по lifecycle с источниками.`

Что проверить в процессе:
- `@rag memory` отражает новую цель и новые ограничения;
- в ответах 4-8 и 10 сохраняется фокус на lifecycle/CLI;
- на 9-м вопросе возможен fallback:
  - `Не знаю по текущему контексту. Уточните вопрос.`;
- после нерелевантного вопроса система возвращается к основной цели без потери контекста.

### Критерии прохождения

- в каждом релевантном ответе есть `Sources` и `Quotes`;
- `@rag memory` показывает накопление `goal/constraints/terms/clarified`;
- ассистент удерживает цель между 10+ сообщениями;
- при слабом контексте корректно срабатывает режим `Не знаю...`;
- после fallback диалог продолжает работать в целевом контексте.

### Как интерпретировать результат демо

Частые ответы `Не знаю по текущему контексту. Уточните вопрос.` в этом сценарии **частично ожидаемы**, но есть границы нормы:

- **Нормально**, если fallback срабатывает:
  - на явно вне-доменных вопросах;
  - при реально слабом/шумном retrieval.
- **Подозрительно**, если fallback идет сериями на профильных вопросах (`day_21`/`day_23`/`day_24`, `TaskState`, `MemoryService`) и источники выглядят нерелевантными.

Практическая интерпретация воспроизведенного диалога:

- Поведение fallback как механизма anti-hallucination — корректное.
- Но слишком частый fallback на профильных вопросах указывает на слабое качество retrieval для текущего индекса/порога в конкретной сессии.
- Это считается **допустимым результатом демо**, если:
  - память задачи (`@rag memory`) накапливается и влияет на ответы;
  - после fallback ассистент способен вернуться к теме и дать grounded-ответ с источниками.

### Типичные причины частого fallback в демо

- слишком общий или шумный индекс (нерелевантные чанки попадают в top-k);
- слишком строгий порог `--rag-min-context-score`;
- команды и запрос в одной строке (например, `@rag memory ...`) вместо двух отдельных сообщений;
- накопление шумных уточнений в памяти задачи.

### Как запускать демо устойчивее

```bash
python3 llm_cli.py --chat --rag \
  --rag-index-path "homeworks/artifacts/day_21/index_structured.json" \
  --rag-top-k-before 12 \
  --rag-top-k-after 6 \
  --rag-similarity-threshold 0.12 \
  --rag-min-context-score 0.18
```

Рекомендации:
- отправляйте `@rag memory` отдельной строкой;
- при серии fallback сначала дайте уточнение по области поиска, затем повторите вопрос;
- оценивайте прохождение сценария по совокупности критериев (memory + возврат в тему + источники), а не только по числу fallback-ответов.
